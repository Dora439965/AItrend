# AI Trend & Vibe Coding Discovery Page PRD

## 1. 项目概述

### 1.1 产品名称

AI Trend & Vibe Coding Discovery

### 1.2 产品定位

一个面向 AI 从业者、开发者、产品经理、创业者和对 vibe coding 感兴趣的新手用户的网站页面，帮助用户快速发现当前 GitHub 上最热门的 AI 开源项目、过去一周 AI 行业高讨论度新闻，并根据个人背景推荐适合上手实践的开源项目。

### 1.3 核心价值

- 帮助用户每天快速了解 AI 开源生态中正在变热的项目。
- 聚合过去一周 AI 行业的重要新闻，优先关注 OpenAI、Anthropic、Google、Meta、Microsoft、xAI、Mistral、Perplexity 等公司。
- 降低 vibe coding 入门门槛，让不同技术水平的用户都能找到适合自己尝试、学习或二次开发的开源项目。

### 1.4 目标用户

- AI 产品经理：想快速了解技术趋势和可借鉴的产品形态。
- 开发者：想寻找值得学习、贡献或二次开发的 AI 项目。
- 创业者/独立开发者：想寻找可组合、可落地的开源项目灵感。
- 编程新手：想通过 vibe coding 尝试构建自己的 AI 应用。
- 研究者/技术观察者：想追踪开源生态与行业新闻。

## 2. 需求细化

### 2.1 模块一：每日 GitHub AI 热门项目榜

#### 2.1.1 功能目标

每天自动更新截止当前时间 GitHub 上热度最高的前 10 个 AI 相关项目、应用、Agent、MCP、skill、工具库、模型工程项目、提示词工程项目等。

#### 2.1.2 项目范围

可纳入榜单的项目类型包括：

- AI 应用：聊天机器人、AI 搜索、AI 办公、AI 编程助手、AI Agent 应用。
- 开发框架：Agent 框架、RAG 框架、LLM 应用开发框架、模型部署框架。
- 工具与插件：MCP server、Codex/Claude Code skill、浏览器插件、CLI 工具。
- 模型相关：微调、推理、评测、数据处理、模型压缩、模型服务。
- 创意项目：AI 图像、音频、视频、游戏、自动化 workflow。

#### 2.1.3 热度评分建议

热门程度不只按总 stars 排序，而应使用综合热度分：

```
Hot Score =
  35% stars 增长
  20% forks 增长
  15% issues / PR 活跃度
  15% discussion / social mentions
  10% 最近 commit 活跃度
  5% 项目成熟度与可用性
```

建议优先计算近 24 小时、近 7 天两个窗口：

- 24h 增长：衡量当天爆发趋势。
- 7d 增长：过滤短期噪声，判断持续热度。

#### 2.1.4 展示信息

每个项目卡片建议包含：

- 项目名称和 GitHub 链接。
- 作者/组织名称。
- 一句话简介。
- 项目类型标签，例如 Agent、RAG、MCP、Coding、Image、Video、CLI。
- 当前 stars、24h stars 增长、7d stars 增长。
- forks、最近更新时间、主要语言。
- 热度原因摘要，例如“近 24 小时 stars 增长显著”“被多个开发者社区讨论”“由知名团队发布”。
- 适用人群标签，例如新手友好、适合二次开发、适合学习架构、适合产品灵感。

#### 2.1.5 数据来源建议

- GitHub Search API / GraphQL API。
- GitHub Trending 页面抓取或第三方 trending 数据源。
- Hacker News、Reddit、X、Product Hunt、Lobsters 等讨论源，可作为讨论度补充。
- 项目 README、release、issue、discussion 数据。

### 2.2 模块二：过去一周 AI 热点新闻

#### 2.2.1 功能目标

展示过去 7 天内 AI 领域讨论度较高、影响较大的新闻，优先展示大公司、重要产品发布、模型发布、融资/收购、监管政策、研究突破、开源发布和开发者生态变化。

#### 2.2.2 新闻优先级规则

优先展示：

- OpenAI、Anthropic、Google DeepMind、Meta AI、Microsoft AI、xAI、Mistral、Perplexity、NVIDIA 等公司的重大新闻。
- 新模型、新 API、新产品、新价格策略、新开发者工具。
- 对开发者生态有直接影响的新闻。
- 社区讨论度高但仍有可信来源支撑的事件。

降低权重：

- 无明确来源的传闻。
- 重复转载但缺少新增信息的新闻。
- 与 AI 关系弱的泛科技新闻。

#### 2.2.3 展示信息

新闻卡片建议包含：

- 新闻标题。
- 来源媒体/官方博客。
- 发布时间。
- 相关公司或主题标签。
- 3-5 行摘要。
- 为什么重要。
- 相关链接。
- 讨论热度指标，例如社媒提及、HN 评论数、Reddit 评论数、X 转发量等。

#### 2.2.4 新闻分类

建议支持以下筛选：

- 大公司动态。
- 模型与研究。
- 开发者工具。
- 开源项目。
- AI 产品与应用。
- 融资与商业。
- 政策与安全。

### 2.3 模块三：Vibe Coding 开源项目推荐

#### 2.3.1 功能目标

为想尝试 vibe coding 的用户推荐适合上手的 GitHub 开源项目，并在用户选择某个项目后，生成一份结构化的上手说明文档。

#### 2.3.2 用户画像采集

推荐前应通过问卷或对话式表单收集以下信息：

- 用户背景：产品、设计、运营、学生、前端、后端、全栈、数据、研究、创业者等。
- 代码水平：零基础、能看懂少量代码、会写简单脚本、能独立开发、资深开发者。
- 开发能力：前端、后端、Python、JavaScript/TypeScript、移动端、数据处理、模型工程等。
- 想做什么：学习、改造项目、做作品集、上线小产品、创业验证、自动化个人工作流。
- 兴趣方向：AI Agent、RAG、AI 搜索、AI 编程、图像生成、视频、语音、知识库、浏览器自动化、效率工具、教育、金融、医疗、游戏等。
- 时间投入：30 分钟、半天、1 天、1 周、长期项目。
- 运行环境偏好：本地运行、Web 应用、CLI、无需复杂部署、可 Docker 部署。
- API 成本接受度：免费优先、可用少量 API 预算、可接入商业模型 API。

#### 2.3.3 推荐逻辑

推荐模型应综合考虑：

- 项目难度与用户代码水平是否匹配。
- README 是否清晰，是否有安装说明和示例。
- 最近是否仍在维护。
- issue 数量和社区活跃度。
- 技术栈是否适合用户。
- 是否容易跑起来。
- 是否有清晰可改造的功能点。
- 是否适合生成一个可展示的小作品。

#### 2.3.4 推荐结果展示

每个推荐项目展示：

- 项目名称、GitHub 链接。
- 项目简介。
- 核心功能。
- 技术栈。
- 推荐理由。
- 适合用户的原因。
- 上手难度：入门、进阶、挑战。
- 预计上手时间。
- 可能的改造方向。
- 风险提示，例如依赖较重、需要 API Key、文档不完整、部署复杂。

#### 2.3.5 项目上手说明文档

用户选择项目后，系统生成一份说明文档，建议包含：

- 项目是什么。
- 适合你从哪里开始。
- 环境准备。
- 本地运行步骤。
- 关键目录结构解释。
- 核心运行流程。
- 第一次建议修改的功能。
- 常见报错与排查。
- 适合继续探索的 issue 或功能点。
- 7 天实践计划。
- 可以做成作品集的展示方式。

## 3. 页面信息架构

### 3.1 页面结构

建议采用单页 Dashboard 结构，而不是营销页。首屏直接呈现核心内容：

1. 顶部导航与更新时间。
2. 今日 GitHub AI 热榜。
3. 本周 AI 新闻。
4. Vibe Coding 项目推荐器。
5. 项目详情 / 上手文档生成面板。

### 3.2 核心布局

桌面端建议：

- 左侧主内容：GitHub 热榜和新闻流。
- 右侧侧栏：筛选器、更新时间、推荐器入口、用户画像摘要。
- 下方或抽屉面板：项目详情和上手文档。

移动端建议：

- 顶部使用分段 tabs：热榜、新闻、推荐。
- 项目详情使用底部弹层或独立详情页。

### 3.3 关键交互

- 切换榜单窗口：今日、近 7 天、近 30 天。
- 筛选项目类型：Agent、RAG、Coding、MCP、Image、Video、Infra 等。
- 点击项目卡片查看详情。
- 收藏项目。
- 对新闻按公司/主题筛选。
- 填写 vibe coding 用户画像。
- 生成推荐项目列表。
- 选择项目后生成上手文档。
- 导出上手文档为 Markdown 或复制到剪贴板。

### 3.4 基础交互逻辑

#### 3.4.1 全局导航与模块切换

页面采用单页 Dashboard 结构，用户可以在同一页面内完成浏览、筛选、推荐和文档生成。

桌面端建议使用顶部导航加页面内锚点：

- 顶部导航包含“AI 热榜”“AI 新闻”“项目推荐”三个入口。
- 点击导航项后，页面平滑滚动到对应模块，并高亮当前模块。
- 当用户滚动页面时，顶部导航同步高亮当前所在模块。
- 顶部右侧固定展示“最后更新时间”和“手动刷新”按钮。
- 手动刷新后，页面展示轻量 loading 状态，刷新成功后显示“已更新至 HH:mm”。

移动端建议使用顶部分段 tabs：

- tabs 包含“热榜”“新闻”“推荐”。
- 用户点击 tab 后切换模块内容，不需要长页面滚动。
- 当前模块内的筛选条件和滚动位置应尽量保留，避免用户来回切换时丢失上下文。

模块之间的联动规则：

- 用户在热榜模块点击“适合我吗”，可带着该项目进入项目推荐模块，系统自动把项目放入候选详情区。
- 用户在新闻模块点击与开源项目相关的新闻时，如果系统识别到相关 GitHub 项目，可以提供“查看相关项目”入口。
- 用户在推荐模块完成画像后，热榜项目卡片可以额外显示“与你匹配”标签或匹配分。

#### 3.4.2 模块一：GitHub AI 热榜交互

用户进入页面后，默认看到“今日 GitHub AI 热榜 Top 10”。每个项目以可扫描的榜单卡片展示。

用户可执行的操作：

- 切换时间范围：今日、近 7 天、近 30 天。
- 切换排序维度：综合热度、stars 增长、forks 增长、讨论度、最近更新。
- 使用标签筛选：Agent、RAG、Coding、MCP、Image、Video、Infra、Tool、Model。
- 使用语言筛选：Python、TypeScript、JavaScript、Go、Rust 等。
- 点击项目卡片查看项目详情。
- 点击 GitHub 图标打开项目原始 GitHub 页面。
- 点击收藏按钮临时收藏项目。
- 点击“不感兴趣”降低同类项目在推荐中的权重。
- 点击“加入推荐对比”把项目加入右侧对比栏。

项目卡片交互状态：

- 默认状态：展示排名、项目名、简介、标签、stars 增长、hot score。
- hover 状态：展示更多快捷操作，例如查看详情、打开 GitHub、加入对比。
- loading 状态：使用骨架屏展示榜单占位。
- 空状态：当筛选后没有结果时，提示用户放宽筛选条件，并提供“一键清除筛选”。
- 错误状态：当 GitHub 数据加载失败时，展示最近一次缓存数据和失败提示。

项目详情建议使用右侧抽屉，而不是居中弹窗：

- 桌面端：点击项目卡片后，从右侧打开详情抽屉，占屏宽约 40%-50%。
- 移动端：点击项目卡片后，从底部打开全屏详情页或底部弹层。
- 抽屉顶部展示项目名、owner、GitHub 链接、收藏按钮和关闭按钮。
- 抽屉主体展示项目简介、核心功能、技术栈、热度指标、热度原因、README 摘要、适合人群、风险提示。
- 抽屉底部固定操作区，包含“查看 GitHub”“加入推荐对比”“生成上手建议”。
- 点击遮罩、关闭按钮或按 Esc 可关闭抽屉。
- 关闭抽屉后保留用户当前筛选条件和榜单滚动位置。

#### 3.4.3 模块二：AI 新闻交互

新闻模块默认展示过去 7 天内讨论度较高的 AI 新闻，按重要性和时间综合排序。

用户可执行的操作：

- 切换新闻时间范围：24 小时、3 天、7 天。
- 按公司筛选：OpenAI、Anthropic、Google、Meta、Microsoft、xAI、Mistral、NVIDIA 等。
- 按主题筛选：模型发布、产品发布、开发者工具、开源、融资、政策、安全、研究。
- 切换排序：综合重要性、最新发布、讨论度最高。
- 点击新闻卡片查看详情。
- 点击来源链接打开原文。
- 点击“相关项目”查看与该新闻相关的 GitHub 项目。
- 点击收藏或稍后阅读。

新闻卡片交互状态：

- 默认状态：展示标题、来源、发布时间、摘要、公司标签、主题标签、讨论度。
- hover 状态：展示快捷按钮，例如查看详情、打开原文、收藏。
- 已读状态：用户打开详情后，标题颜色或标记变弱，帮助区分已读新闻。
- 多来源状态：同一事件来自多个来源时，卡片显示“多来源报道”，详情中聚合展示来源列表。
- 空状态：筛选无结果时提示用户调整公司或主题筛选。

新闻详情建议使用居中弹窗或右侧抽屉，两种方案可按信息密度选择：

- 如果新闻详情较短，使用居中弹窗，适合快速阅读。
- 如果需要展示多来源、相关项目、时间线和讨论链接，使用右侧抽屉。

新闻详情内容建议包含：

- 标题、来源、发布时间、原文链接。
- 3-5 行摘要。
- “为什么重要”说明。
- 相关公司、产品、模型或人物标签。
- 多来源链接列表。
- 相关 GitHub 项目或开源工具。
- 讨论入口，例如 Hacker News、Reddit、X 或官方论坛链接。

新闻详情交互规则：

- 点击原文链接默认新标签页打开，不打断当前页面状态。
- 点击相关项目时，打开项目详情抽屉；如当前新闻详情为弹窗，可先关闭新闻弹窗再打开项目抽屉。
- 如果新闻与某个热榜项目相关，项目详情中也应展示“相关新闻”反向入口。

#### 3.4.4 模块三：Vibe Coding 推荐器交互

推荐器采用分步骤表单或对话式问卷，降低新手填写压力。默认入口可以是右侧侧栏卡片，也可以是独立模块中的主操作区。

推荐流程建议分为四步：

1. 选择用户背景和代码水平。
2. 选择兴趣方向、目标和时间投入。
3. 选择技术栈、运行环境和 API 成本偏好。
4. 生成推荐项目并展示匹配原因。

用户可执行的操作：

- 使用单选、多选、滑块或标签选择完成画像。
- 点击“跳过”跳过非必填问题。
- 点击“重新生成”在同一画像下刷新推荐结果。
- 点击“调整画像”返回问卷修改答案。
- 点击推荐项目查看详情。
- 点击“生成上手文档”生成该项目的 Markdown 指南。
- 点击“复制文档”或“导出 Markdown”。

推荐问卷交互状态：

- 未开始状态：展示简短说明和“开始推荐”按钮。
- 填写中状态：展示当前步骤、进度条、上一步、下一步。
- 生成中状态：展示推荐生成 loading，并提示正在匹配项目难度、技术栈和兴趣方向。
- 结果状态：展示 3-5 个推荐项目，每个项目包含匹配分、推荐理由、上手难度和预计时间。
- 无结果状态：提示用户放宽技术栈、时间投入或运行环境限制。

推荐结果卡片交互：

- 默认展示项目简介、匹配分、适合原因、难度、预计上手时间。
- 点击卡片打开项目详情抽屉。
- 点击“为什么适合我”展开推荐解释。
- 点击“我不喜欢这个”后，系统移除该项目并尝试补充一个新推荐。
- 点击“生成上手文档”后，在页面右侧或下方打开文档预览面板。

上手文档生成交互：

- 生成前展示项目、用户画像和文档结构预览。
- 生成中展示分段 loading，例如“读取 README”“分析目录结构”“整理运行步骤”“生成 7 天计划”。
- 生成完成后展示 Markdown 预览。
- 文档预览支持章节折叠、复制全文、复制单个代码块、导出 Markdown。
- 如果项目 README 缺少安装说明，文档中应明确提示“不确定项”，避免编造。
- 用户可以点击“重新生成文档”，并选择偏好：更适合新手、更偏工程实践、更偏产品改造、更短版本。

#### 3.4.5 弹窗、抽屉与页面状态规范

为了保持页面专业和高效，详情类内容建议优先使用抽屉，确认类或短信息使用弹窗。

使用抽屉的场景：

- GitHub 项目详情。
- 推荐项目详情。
- 上手文档预览。
- 包含多来源和相关项目的新闻详情。

使用弹窗的场景：

- 简短新闻详情。
- 确认清空筛选条件。
- 确认离开未保存的推荐问卷。
- 提示复制成功或导出成功。

基础交互规范：

- 所有弹窗和抽屉都必须有明确关闭按钮。
- 点击遮罩可关闭非编辑状态弹窗。
- 用户正在填写问卷或生成文档时，关闭前需要二次确认。
- 弹窗打开后，键盘焦点应进入弹窗内部。
- 按 Esc 可关闭普通弹窗或抽屉。
- 页面刷新或模块切换时，应保留用户最近一次筛选条件。
- 所有外部链接默认新标签页打开。
- 所有生成内容必须展示生成时间和依赖的数据更新时间。

#### 3.4.6 典型用户路径

路径一：趋势浏览型用户

1. 用户进入页面，查看今日 AI 热榜。
2. 用户切换到“近 7 天”，筛选 Agent 项目。
3. 用户点击某个项目卡片，右侧打开项目详情抽屉。
4. 用户查看热度原因和 README 摘要。
5. 用户点击 GitHub 链接跳转到原项目。

路径二：新闻追踪型用户

1. 用户进入新闻模块。
2. 用户筛选 OpenAI 和 Anthropic。
3. 用户点击某条新闻查看详情。
4. 用户从详情中打开原文或相关讨论。
5. 用户点击“相关项目”，跳转到项目详情抽屉。

路径三：vibe coding 新手用户

1. 用户点击“找到适合我的项目”。
2. 用户完成 4 步画像问卷。
3. 系统返回 3-5 个推荐项目。
4. 用户展开“为什么适合我”查看推荐解释。
5. 用户选择一个项目并生成上手文档。
6. 用户复制 Markdown 文档，开始本地实践。

## 4. 产品体验原则

- 信息密度适中，优先帮助用户扫读和比较。
- 不做空泛首页，用户进入页面后立即看到最新榜单和新闻。
- 强调“为什么热门”和“为什么适合你”，而不只是罗列链接。
- 对新手友好，但不牺牲专业用户需要的指标。
- 所有自动生成内容应标明数据更新时间和来源。
- 新闻摘要和项目推荐需要保留原始链接，方便用户核验。

## 7. 视觉与交互设计建议

### 7.1 风格方向

页面应采用专业、清晰、偏工具型的 Dashboard 风格。避免营销页式的大 hero 和装饰性内容，重点放在信息比较、筛选和行动。

### 7.2 组件建议

- 顶部状态栏：显示日期、数据更新时间、刷新按钮。
- 榜单卡片：排名、项目名、指标、标签、热度原因。
- 新闻列表：标题、来源、摘要、主题标签。
- 推荐器：分步骤表单或对话式问卷。
- 项目详情抽屉：README 摘要、指标、适合人群、风险。
- 文档预览面板：Markdown 格式，可复制或导出。

### 7.3 色彩建议

使用中性背景搭配少量高识别强调色：

- 背景：浅灰或近白。
- 文本：深灰。
- 强调色：蓝绿色或电光蓝，用于热度、增长、行动按钮。
- 警示：橙色用于风险提示。
- 成功：绿色用于“易上手”“文档清晰”等正向标签。

## 9. 开发计划

### 阶段一：需求确认与原型设计，2-3 天

- 明确目标用户和 MVP 范围。
- 完成信息架构。
- 输出低保真原型。
- 确定数据源和更新策略。

### 阶段二：数据采集与评分模型，4-6 天

- 接入 GitHub API。
- 建立项目热度评分。
- 接入新闻源或 RSS。
- 建立新闻去重和重要性评分。
- 存储每日快照数据。

### 阶段三：前端页面开发，5-7 天

- 实现 Dashboard 布局。
- 实现热榜、新闻、筛选、详情面板。
- 实现推荐器表单。
- 实现响应式布局。

### 阶段四：推荐与文档生成，4-6 天

- 建立用户画像数据结构。
- 实现项目匹配评分。
- 生成推荐理由。
- 生成项目上手 Markdown 文档。
- 增加复制和导出功能。

### 阶段五：测试与发布，3-5 天

- 数据质量测试。
- 推荐准确性人工评估。
- 前端兼容性测试。
- 移动端体验测试。
- 部署上线。

## 10. 风险与应对

### 10.1 GitHub API 限流

应对方式：

- 使用缓存和定时任务。
- 减少实时请求。
- 优先存储每日快照。
- 必要时使用 GitHub token 提高限额。

### 10.2 热度评分不准确

应对方式：

- 同时展示原始指标。
- 保留人工调权空间。
- 支持用户反馈“不相关”或“不热门”。

### 10.3 新闻来源质量不稳定

应对方式：

- 优先官方博客和可信媒体。
- 对重复新闻做聚合。
- 明确展示来源与发布时间。

### 10.4 推荐结果过泛

应对方式：

- 增加用户画像维度。
- 提高项目文档质量、难度匹配的权重。
- 为推荐理由提供可解释性。
